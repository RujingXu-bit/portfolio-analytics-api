# 项目执行规则

本文件是本仓库内所有 Codex task、agent 和人工开发工作的长期执行契约。开始任何任务前，必须先阅读本文件和根目录的 `PROJECT_PLAN.md`。若任务描述与项目计划存在冲突，应先指出冲突，不得自行扩大范围。

## 1. 项目目标

构建一个求职展示级的 AI 驱动投资组合分析后端：用户记录投资组合与交易，系统获取市场数据，计算可解释、可复现的金融指标，并可选择性地生成基于结构化指标的风险摘要。

项目优先级依次为：

1. 金融计算正确且口径透明。
2. 数据所有权、金额精度和错误处理可靠。
3. 核心流程可测试、可复现、可本地启动。
4. 架构能够清楚解释，但不为展示而过度设计。
5. AI、第二市场数据源和前端属于核心系统稳定后的扩展。

## 2. 开始任务前

每个任务必须执行以下步骤：

1. 阅读 `AGENTS.md` 和 `PROJECT_PLAN.md`。
2. 确认本次只处理一个明确的 Task ID，例如 `W2.3`。
3. 检查工作区状态，保留用户或其他任务留下的无关修改。
4. 阅读任务涉及的现有代码、测试和架构决策后再修改。
5. 如果任务依赖尚未完成，不伪造依赖；记录阻塞并停止相关实现。

除非任务明确要求，不顺手加入前端、微服务、消息队列、复杂部署或其他计划外功能。

### 2.1 里程碑审查基线

开始任何里程碑 Review 前，必须完成以下基线检查：

1. 执行 `git fetch --prune origin`，刷新远端引用并清理已删除的远端分支引用；若同步失败，停止审查并报告。
2. 确认当前分支和本次目标审查分支；PR 已合并时默认审查最新 `origin/main`，不得仅依据未同步的本地 `main`。
3. 确认本地 `HEAD` 与目标远端引用完全一致；不一致时先完成安全同步，不得基于旧提交给出审查结论。
4. 记录本次审查基线 commit hash。
5. 确认工作区干净；若存在未提交修改，停止审查并报告，不得擅自丢弃、覆盖或混入审查结果。
6. 只有分支、commit 和工作区状态全部确认后，才能开始读取实现、运行测试和更新里程碑状态。

每份里程碑审查报告必须明确包含：

```text
Review baseline: <branch>@<commit-hash>
HEAD == <target-remote-ref>: yes
Worktree clean: yes
```

如果审查对象是尚未合并的 PR，应将 `<branch>` 和 `<target-remote-ref>` 替换为该 PR 的明确分支与远端引用；不得混用其他工作树、旧分支或缓存的远端状态作为验收证据。

## 3. 技术基线

- Python：使用项目固定的 Python 版本，以 `.python-version` 和 `pyproject.toml` 为准。
- Web：FastAPI。
- 数据库：PostgreSQL 16、SQLAlchemy 2.x、Alembic。
- 缓存：Redis 7。
- 包与环境管理：uv；提交 `uv.lock`，不提交 `.venv`。
- 质量工具：Ruff、mypy、pytest、pytest-cov。
- 本地基础设施：Docker Compose。
- 统一命令入口：根目录 `Makefile`。
- CI：GitHub Actions。

不要混用 pip、Poetry 和 uv 管理项目依赖。新增依赖必须说明用途，并使用 `uv add` 或 `uv add --dev` 更新 `pyproject.toml` 与 `uv.lock`。

## 4. 架构约束

采用模块化单体。保持 API、应用服务、领域计算和基础设施适配器之间的边界，但避免没有实际价值的抽象层。

推荐职责划分：

```text
src/
├── api/              # FastAPI 路由、依赖注入、HTTP schema
├── application/      # 用例编排、权限与事务边界
├── domain/           # 实体、值对象、金融纯函数、协议
├── infrastructure/   # SQLAlchemy、Redis、市场数据和 LLM 适配器
├── core/             # 配置、日志、安全和公共异常
└── main.py            # 应用组装入口
```

具体目录可以在 W1.1 中微调，但必须遵守以下规则：

- 路由不得直接包含金融计算、SQL查询或第三方 API 细节。
- 金融指标优先实现为无网络、无数据库、无当前时间依赖的纯函数。
- 数据访问通过 Repository 边界组织，领域层不得依赖 SQLAlchemy 模型。
- 市场数据通过 `MarketDataProvider` 抽象访问。
- Provider 向内部返回项目自己的 `PriceBar` 类型，不向业务层泄漏 Pandas `DataFrame` 或供应商响应结构。
- yfinance 是阻塞式适配器，异步请求路径中必须在线程池或后台任务中调用。
- 不拆分微服务；除非计划被明确修改，否则不引入 Celery、Kafka 等基础设施。

## 5. 金融与数据规则

- 数据库中的金额、价格、数量和费用使用 PostgreSQL `NUMERIC`，Python 业务对象使用 `Decimal`。
- 禁止使用二进制 `float` 持久化货币值。
- 统计计算可在边界处显式转换为 `float64`，但必须测试误差容忍度，并且不得用转换后的值覆盖原始金额。
- 默认使用 adjusted close 计算历史收益；若 Provider 不支持，必须明确标记口径。
- 默认采用简单日收益率；年化周期默认 252，但必须作为参数或配置暴露。
- 无风险利率不得散落硬编码；其数值、日期和假设必须出现在分析结果的 methodology 中。
- 缺失价格、重复日期、非交易日、时区、拆股和分红必须有明确处理规则。
- 指标响应必须包含 `as_of` 和 methodology，保证结果可解释、可复现。
- 不实现或宣传股票价格预测、保证收益或自动买卖建议。

第一版核心指标为：

- 简单收益率。
- 年化波动率。
- 最大回撤。
- Sharpe Ratio。

## 6. 市场数据与缓存规则

- 单元测试不得访问真实网络；使用 `FakeMarketDataProvider` 和固定 fixture。
- 真实 Provider 测试必须标记为可选的 contract/integration test，普通 CI 不依赖第三方服务可用性。
- V1 只要求 Fake Provider 加一个真实 Provider；第二真实 Provider 是扩展任务。
- 所有外部调用必须设置超时，并将 429、5xx、无数据和无效 symbol 映射为稳定的内部错误。
- 重试必须有次数上限，只对适合重试的错误执行，不重试确定性的 4xx。
- 缓存 TTL 根据数据新鲜度设置：报价使用短 TTL，已闭市的历史数据使用较长 TTL。
- 返回旧缓存时必须标记 `stale`，不得把过期数据伪装成实时数据。
- API Key、Token 和凭据只从环境变量读取；仓库仅提交无秘密的 `.env.example`。

## 7. AI 模块规则

- LLM 只解释后端已经计算出的结构化指标，不负责计算关键金融数值。
- 输出范围限于风险摘要、集中度说明、历史情景和数据限制。
- 不生成明确的买入、卖出或收益保证。
- LLM 超时、限额或解析失败不得导致核心 analytics 接口失败。
- 必须提供确定性的规则摘要作为回退。
- 保存分析快照时记录模型、提示词版本、生成时间和输入指标摘要，不保存秘密。

## 8. API、数据库与安全规则

- 所有输入使用 Pydantic schema 验证；错误响应保持统一结构。
- 创建交易需要幂等键或可唯一识别的 `external_id`，防止重复导入。
- 用户只能访问自己拥有的 Portfolio、Transaction 和 AnalysisSnapshot。
- 权限校验必须发生在应用服务或统一依赖中，不依赖前端隐藏资源。
- 密码只保存安全哈希，不记录明文密码、JWT、API Key 或完整敏感请求。
- 数据库结构变化必须通过 Alembic migration 完成。
- migration 必须能从空数据库升级到最新版本；不得只修改 ORM 模型。
- 删除数据库卷、清空表或执行不可逆迁移前必须获得明确授权。

## 9. 测试与质量门槛

测试分层：

- `tests/unit`：纯函数、值对象、Fake Provider；必须快速且无外部依赖。
- `tests/integration`：PostgreSQL、Redis、Repository 和 API 完整交互。
- `tests/contract`：真实外部 Provider；默认不在普通 CI 中运行。

测试要求：

- 金融指标必须覆盖正常、空数据、单点数据、零波动、持续下跌、缺失日期等边界。
- 金额比较优先使用精确 `Decimal`；统计值使用明确的近似容差。
- 数据库测试必须隔离，不能依赖测试执行顺序。
- 认证测试必须证明用户 A 无法访问用户 B 的资源。
- 只有测试确实依赖系统时钟时才使用时间冻结工具。
- 不为追求覆盖率数字编写无意义断言；关键领域分支必须被覆盖。
- 性能改善和缓存收益只能报告实测数据，禁止预先编造百分比。

提交前，按改动范围运行最小充分验证。`Makefile` 建立后优先使用：

```bash
make lint
make typecheck
make test
```

涉及数据库、Redis 或 API 时还需运行对应集成测试。完整交付前运行 `make check` 和覆盖率测试。

## 10. 文档与决策记录

- `PROJECT_PLAN.md` 是任务状态和优先级的唯一权威来源。
- `README.md` 面向使用者和面试官，不承担内部任务追踪。
- 重要且难以逆转的选择写入 `docs/decisions.md`，说明背景、决定和取舍。
- 架构变化同步更新 `docs/architecture.md`。
- API 或金融口径变化必须同步更新相关文档与测试。
- 文档中的性能数字、覆盖率和功能状态必须与当前仓库事实一致。

## 11. 完成任务后的要求

只有满足任务验收标准后，才能在 `PROJECT_PLAN.md` 中将对应复选框改为 `[x]`。部分完成保持 `[ ]`，并在进度日志中写明剩余工作。

每个完成的任务应：

1. 运行相应测试和质量检查。
2. 更新受影响的文档。
3. 更新 `PROJECT_PLAN.md` 的任务状态和进度日志。
4. 汇报修改文件、验证命令、结果及未解决风险。
5. 不声称未运行的测试通过，也不声称未测量的性能改善。

## 12. 项目级完成定义

V1 只有同时满足以下条件才视为完成：

- 核心 Portfolio 和 Transaction 流程可用。
- PostgreSQL migration 可从空库执行。
- 一个真实市场数据 Provider 可用，且 Fake Provider 支持稳定测试。
- 四个核心指标经过边界测试并返回 methodology。
- Redis 缓存、超时和错误降级按设计工作。
- JWT 认证和资源所有权测试通过。
- LLM 可选模块失败时核心分析仍可用。
- Ruff、mypy、单元测试和集成测试在 CI 中通过。
- Docker Compose、README 和演示流程在干净环境验证成功。
- 所有公开性能数字均来自可复现测试。
